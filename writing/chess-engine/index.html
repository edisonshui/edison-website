building a chess engine that actually beats me

May 12, 2025 8 min read

I kept wondering why my chess AI would blunder obvious pieces. Turns out it was evaluating 8 million nodes a second and still playing like my little cousin. The problem wasn't search depth — the problem was that I'd written my evaluation function during a 2 AM session and it thought a passed pawn in the center was worth about as much as the queen.

This is the story of three weeks, one algorithm, and the slow realization that "minimax" is a five-minute explanation and a three-week implementation.

The algorithm, the way I wish someone had explained it

Minimax is simple in theory: you're trying to maximize your score, your opponent is trying to minimize it. At every move, you pick the best outcome assuming your opponent also plays perfectly. Recursively. Down to some depth.

The version in every textbook makes it look obvious. Here's what they leave out: at depth 5, a standard chess position has about 69 billion nodes. You can't evaluate all of them. You need pruning.

Alpha-beta pruning

Alpha-beta is an optimization that cuts branches of the game tree you can prove will never be reached. If you've already found a move that guarantees a score of +2, and you're evaluating a branch that can only give your opponent at most +1, you stop — there's no reason to go deeper.

In the best case, alpha-beta halves the effective depth. In practice, with good move ordering, you can get close to that ceiling.

function minimax(board, depth, alpha, beta, maximizing) {
  if (depth === 0) return evaluate(board);

  const moves = board.getLegalMoves();
  if (maximizing) {
    let best = -Infinity;
    for (const move of moves) {
      board.makeMove(move);
      const score = minimax(board, depth - 1, alpha, beta, false);
      board.undoMove(move);
      best = Math.max(best, score);
      alpha = Math.max(alpha, best);
      if (beta <= alpha) break; // prune
    }
    return best;
  } else {
    // symmetric for minimizing player
  }
}

The first version I wrote had a bug: I was passing alpha and beta by value, not by reference, so the pruning never actually activated. The engine was running full minimax at depth 5 and taking 8 seconds per move. My friend thought it was "thinking really hard."

What the evaluation function actually does

The evaluation function is where I lost the most time. My first version was: count the material, assign piece values (pawn=1, knight=3, bishop=3, rook=5, queen=9), return the difference.

A passed pawn on e5 is not the same as a passed pawn on a2. An open file means nothing if you have no rooks. I knew this. My engine did not.

The fix was piece-square tables — lookup tables that give each piece a positional bonus depending on where it sits on the board. Knights are worth more in the center. Rooks want open files. Kings want to hide in the corner until the endgame, then centralize. Once I added these, the engine started making moves that at least looked intentional.

The Web Worker problem

The browser froze every time the AI was thinking. Obviously. Running a recursive search on the main thread blocks the UI. I needed to offload the search to a Web Worker.

The tricky part was serializing the board state. My Board class had methods — you can't pass class instances through postMessage with their prototypes intact. I ended up converting everything to a plain object representation (a flat array of piece codes) and deserializing on the worker side. Not elegant, but it works.

The bigger win: now you can watch the AI "thinking" while the board stays responsive. A small progress indicator and the elapsed time display. It changed how the game felt — suddenly it had weight.

Where I am now

The engine handles all legal moves, castling, en passant, promotion, and threefold repetition detection. At depth 5 with alpha-beta and a small opening book, it plays around 1300 Elo — which is slightly above my actual rating, which is humbling in a very specific way.

Next: quiescence search (keep searching until the position is "quiet"), then iterative deepening so the engine can always return a best move if it runs out of time.

If you want to play against it, it's on the playground. It'll beat you on hard. I'm sorry in advance.
